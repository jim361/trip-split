# Trip Split Decision History

이 문서는 Trip Split을 기획하면서 상의하고 발전시킨 주요 결정 흐름을 남긴 기록이다.

## 1. 시작점: 강릉 여행 HTML

처음에는 기존 `gangneung_trip_editable_sortable_budget.html`을 기반으로 아이디어를 정리했다. 이 HTML은 강릉 1박 2일 여행 일정, Leaflet 기반 지도 표시, 고정 참여자 중심 정산 기능을 한 파일에 담고 있었다.

초기 아이디어는 이 HTML을 확장해 다음 기능을 가진 앱 또는 웹 서비스로 발전시키는 것이었다.

- 장소 기반 여행 일정 관리
- 지도 핀과 동선 표시
- 참여자 기반 정산
- 영수증 OCR
- 국내 여행 우선
- 이후 해외 여행 확장

## 2. 국내 MVP와 해외 확장

처음부터 해외 여행까지 모두 만들기보다 국내 여행에 집중하기로 했다. 국내 여행에서는 네이버 지도, 원화 정산, 한국어 OCR이 더 적합하다고 판단했다.

해외 여행은 이후 확장으로 두고, Google Maps, 외화/환율, 타임존, 다국어 OCR을 붙일 수 있게 provider 구조를 고려하기로 했다.

결정:

- MVP는 국내 여행 중심
- 해외 여행은 Google Maps 기반 확장
- 여행 생성 시 국내/해외 타입을 둘 수 있게 설계

## 3. 문서화 방식

Kiro에서 사용하던 6개 markdown 문서 구성 방식을 활용하기로 했다.

초기 문서 구성:

- `product.md`

- `design.md`
- `structure.md`
- `tasks.md`

이후 실제 개발 편의성을 위해 task는 기능별 파일로 분리했다.

## 4. 웹/PWA 우선 결정

앱과 웹 중 무엇으로 시작할지 논의했다. 초기에는 앱도 고려했지만, MVP 검증 속도와 공유 편의성을 고려해 웹/PWA로 시작하기로 했다.

결정:

- Vite + React + TypeScript 기반 웹/PWA
- 모바일에서도 사용 가능하게 반응형 설계
- 네이티브 앱은 OCR, 알림, 앱스토어 배포 필요성이 명확해진 뒤 고려

## 5. 서버 없는 공유에서 Firebase 공유로 전환

처음에는 외부 서버 없이 `.trip.json` 파일 내보내기/가져오기로 공유하는 방식을 고려했다. 이 방식은 구현이 쉽고 서버 비용이 없지만, 실시간 공동 편집과 사용자 정보 관리가 어렵다.

이후 Firebase를 백엔드로 사용하기로 하면서 공유 방식이 바뀌었다.

이전 방향:

- 로컬 저장
- `.trip.json` 공유
- 공유 링크는 스냅샷에 가까움

현재 방향:

- Firebase Firestore 저장
- 공유 코드 또는 초대 링크로 여행 세션 참여
- Firestore 실시간 구독으로 공동 편집
- `.trip.json`은 백업/복원/데모 데이터 용도

## 6. 로그인 없음에서 익명 인증 + 선택 Google 로그인으로 발전

초기에는 사용자가 로그인 없이 바로 쓰는 UX를 중요하게 봤다. Firebase를 쓰기로 하면서 "로그인 없음"을 완전히 인증이 없는 상태로 두기보다, 사용자-facing 로그인 화면은 없지만 내부적으로 Firebase Anonymous Auth를 사용하는 방식으로 바꿨다.

이후 Google 로그인도 선택 연결로 추가하는 것이 좋다고 판단했다.

결정:

- 기본 진입은 Anonymous Auth
- 사용자는 로그인 화면 없이 여행 생성 가능
- 선택적으로 Google 로그인 연결
- Google 로그인 시 내 여행 목록, 프로필, 작성자 정보 저장 가능

## 7. 공유 코드와 여행 세션

공유 코드는 단순 조회 키가 아니라 여행 세션 입장 키로 보기로 했다.

기본 흐름:

```text
앱 접속
-> Anonymous Auth uid 발급
-> 여행 생성
-> 공유 코드/초대 링크 생성
-> 친구가 코드 입력 또는 링크 접속
-> members에 등록
-> 같은 tripId의 데이터 실시간 구독
```

MVP 권한은 단순하게 공유 코드나 링크를 가진 사람 모두 편집 가능으로 시작한다.

이후 확장:

- owner
- editor
- viewer
- settlementViewer
- 공유 코드 만료
- 링크 재생성
- 멤버 내보내기

## 8. 네이버 저장 리스트 import 논의

처음에는 사용자가 네이버 지도에 저장한 장소 리스트를 가져올 수 있으면 좋겠다고 생각했다. 실제 샘플 링크 `https://naver.me/xoGa4sgy`를 확인했고, 네이버 공유 리스트가 다음 형태로 리다이렉트되는 것을 확인했다.

```text
https://map.naver.com/p/favorite/sharedPlace/folder/{shareId}
```

또한 네이버 지도 웹앱 내부에서 공유 리스트 장소 목록을 JSON으로 가져오는 엔드포인트가 있음을 확인했다.

```text
https://pages.map.naver.com/save-pages/api/maps-bookmark/v3/shares/{shareId}/bookmarks
```

다만 이 API는 NAVER Cloud Maps 공식 공개 API가 아니라 네이버 지도 웹앱 내부 API로 보인다. 약관, 안정성, 응답 구조 변경 리스크가 있어 MVP 기본 기능으로 삼지 않기로 했다.

결정:

- 네이버 저장 리스트 import는 MVP 제외
- 실험 기능 후보로만 유지
- 기본 장소 추가 흐름은 네이버 장소 검색, 장소 링크 붙여넣기, 직접 입력

## 9. 지도 표시 방향

지도는 실제 내비게이션 경로가 아니라 여행 일정 순서 시각화가 먼저라고 판단했다.

결정:

- 네이버 지도 기반
- 장소가 일정에 연결되면 지도에 표시
- 날짜별 색상 핀
- 일정 순서 번호 핀
- 같은 날짜 장소는 직선 polyline으로 연결
- 실제 도로 경로 계산과 이동 시간은 이후 기능

## 11. OCR 방향

OCR은 있으면 좋은 기능으로 시작했지만, MVP에 포함하기로 했다. 다만 자동 정산까지 바로 연결하지 않고 사용자가 확인하는 흐름을 유지한다.

결정:

- 사용자가 로컬 영수증 이미지 선택
- Cloud Functions에서 CLOVA OCR 호출
- 추출 텍스트 표시
- 금액 후보 추출은 가능하면 시도
- 사용자가 확인한 뒤 지출 등록
- 이미지는 MVP에서 저장하지 않는 방향 우선

텍스트와 총액 후보 중심의 이 결정은 아래 16번 결정에서 수정 가능한 항목 초안, 누락 항목 추가, 항목별 배분과 수동 등록 fallback으로 구체화되었다. 사용자 확인 전에는 정산에 반영하지 않는 원칙은 유지한다.

## 15. 아직 열려 있는 결정

- 공유 코드 만료와 재생성을 어디까지 구현할지
- Firestore 보안 규칙을 members 기준으로 얼마나 엄격하게 둘지
- 네이버 장소 링크 파싱의 안정성을 어떻게 검증할지
- OCR 이미지 최대 크기, 호출 제한, timeout을 어떤 값으로 둘지
- Firebase와 네이버 API의 예산 알림 및 사용량 제한을 어떻게 둘지

다음 사항은 더 이상 열린 결정이 아니다.

- Google 연결은 MVP의 선택 기능이다.
- MVP 영수증 이미지는 Firebase Storage에 영구 저장하지 않는다.
- 네이버 저장 목록 import는 공식 API와 약관 문제가 해결되기 전까지 MVP가 아닌 실험 기능으로만 둔다.
- 사용자가 지정하는 영수증 항목별 분할은 MVP이며, 소비자 자동 추론과 자동 배정은 후속이다.

## 16. 2026-07-12 - 개인 소비 원장과 항목별 분할을 MVP로 확장

기존 정산은 누가 누구에게 얼마를 보내야 하는지는 보여줬지만, 사용자가 이번 여행에서 실제로 무엇에 얼마를 소비했는지는 설명하지 못했다. 같은 지출에서도 결제자와 소비자가 다를 수 있고 메뉴별 소비자가 다르므로, 최종 송금만으로는 개인 경험을 이해하기 어렵다고 판단했다.

결정:

- 개인 정산을 `내가 결제한 금액`, `내가 부담한 금액`, `정산 결과`로 분리한다.
- 정산 결과는 `결제액 - 부담액`으로 계산하며 양수는 받을 금액, 음수는 보낼 금액이다.
- 개인 부담액을 카테고리별로 집계하고 날짜·장소·메뉴 또는 지출 항목별 상세 내역을 제공한다.
- `TripMember`는 인증된 공동 편집 사용자, `Participant`는 정산 대상자로 분리한다.
- 확정 지출은 `payer`, `consumers`, `allocationMethod`, `allocatedAmounts`, `receiptItems`를 가진다.
- MVP 분할은 지출 전체 균등, 항목별 소비자 지정과 공용 항목 균등, 참여자별 직접 금액 입력을 지원한다.
- 할인은 음수, 봉사료는 양수, 기타 조정은 0이 아닌 양수 또는 음수 항목으로 보고 선택한 소비자에게 균등 또는 직접 금액으로 배분한다.
- 모든 KRW 금액은 원 단위 정수로 처리하고 균등 분할 나머지는 화면에 표시된 소비자 순서대로 1원씩 배분한다.
- 정산 snapshot과 송금 완료 상태는 저장하지 않고 확정 지출 원장에서 파생 계산한다.

OCR 결정:

- CLOVA OCR 결과는 저장 전 수정 가능한 초안이다.
- 사용자는 항목명과 금액을 수정하고 누락 항목과 조정을 수동 추가할 수 있다.
- 인식 실패 시 전체 금액 기반 수동 지출 등록으로 전환한다.
- 사용자 확정 전에는 Firestore 원장이나 정산 결과에 반영하지 않는다.
- MVP에서는 Trip Split의 Firestore와 Firebase Storage에 영수증 이미지를 영구 저장하지 않는다. CLOVA OCR 측 처리·보관 고지는 확인된 공급자 정책에 맞춰 별도로 제공한다.

이 결정은 10번의 균등 분할 중심 범위와 11번의 텍스트·총액 중심 OCR 범위를 대체해 구체화한다. 자동 소비자 추론, 고급 품목 분류, 이미지 보관과 비동기 검토 큐는 후속으로 남긴다.

## 17. 2026-07-12 - 모바일 우선 PWA 앱 셸 확정

모바일과 PC에서 서로 다른 정보 구조를 만들면 구현과 학습 비용이 늘고, 여행 중 휴대폰에서 사용하던 흐름을 PC 계획 화면에서 다시 익혀야 한다. 따라서 같은 네 개 목적지와 라우트를 유지하고 PC에서는 화면 영역만 확장하기로 했다.

결정:

- 전역 내비게이션은 `일정`, `지도`, `정산`, `영수증` 네 개로 고정한다.
- 모바일은 고정 하단 내비게이션, PC는 같은 순서와 경로를 확장한 내비게이션을 사용한다.
- 장소 보관함은 메인 메뉴가 아니라 일정 또는 지도 화면의 패널·바텀시트다.
- 모바일 일정은 세로 타임라인, PC 일정은 날짜 목록·타임라인·장소 보관함의 확장 배치다.
- 모바일 지도는 전체 지도와 일정 바텀시트, PC 지도는 일정 사이드 패널과 지도다.
- 모바일 정산은 개인 요약, 최종 정산, 지출 목록 순서이고 PC는 지출 관리와 정산 결과를 다중 열로 배치한다.
- 모바일과 PC 전용 페이지를 복제하지 않고 같은 기능 컴포넌트를 반응형 레이아웃으로 배치한다.

## 18. 2026-07-12 - 3인 소유권과 통합 책임 확정

`정산`, `지도 API와 일정`, `남은 파트`로 나누면 남은 파트에 인증, 공유, 앱 셸, PWA, 백업과 최종 통합이 모두 몰리고 기능 경계가 모호해진다. 도메인 결합도가 높은 기능을 함께 묶고 공통 파일의 최종 책임자를 한 명 두기로 했다.

결정:

| 담당 | 소유 영역 |
| --- | --- |
| 플랫폼·통합 | Vite/Firebase 기반, 인증, 여행 생성·공유·멤버, 앱 셸·라우팅·PWA, 공통 UI, 보안 규칙 통합, 백업, 최종 QA·병합 |
| 정산·영수증 | Participant/Expense/ReceiptItem 모델, 분할·정산 엔진, 개인 소비 화면, 지출 UI, CLOVA OCR Function과 영수증 검토 화면 |
| 장소·일정·지도 | 네이버 장소 검색·링크 파싱·직접 입력, 장소 보관함, 일정 편집, 지도 adapter, 번호 핀·직선 동선 |

- 사용자가 플랫폼·통합 담당이자 제품·데이터 계약·PR 병합의 최종 책임자다.
- OCR은 정산 원장과 검토 UI에 결합되므로 정산·영수증 담당이 소유한다.
- 장소 검색과 링크 파싱은 `Place` 정규화와 지도 표시 계약에 결합되므로 장소·일정·지도 담당이 소유한다.
- 공통 타입과 Firestore 경로 변경은 담당자가 제안하고 세 명이 검토한다.
- Firebase 초기화, Functions 진입점, 공통 라우트, 보안 규칙, dependency와 lockfile은 플랫폼·통합 담당이 최종 병합한다.
- 도메인 작업은 mock repository로 시작하고 플랫폼 기반이 준비되면 Firestore와 외부 API를 연결한다.
- 같은 작업 폴더를 사람이나 AI 작업자가 동시에 편집하지 않고 별도 clone 또는 worktree와 짧은 기능 PR을 사용한다.

이 구조는 세 명의 투입량이 비슷하다는 전제에서 정산·OCR, 장소·일정·지도, 플랫폼·통합의 난이도를 균형 있게 나누고 병합 충돌을 줄이기 위한 결정이다.

## 19. 2026-07-14 - 일정과 지도 화면 통합

여행 중에는 지도 자체를 독립 목적지로 오가기보다 전체 동선을 먼저 보고 바로 아래의 시간표를 이어서 확인하는 흐름이 더 자연스럽다고 판단했다. 주요 내비게이션 수도 줄여 모바일 하단 메뉴의 집중도를 높이기로 했다.

결정:

- 주요 내비게이션은 `일정·지도`, `정산`, `영수증` 세 개로 변경한다.
- 일정 화면 최상단에 축소 지도를 두고 그 아래에 날짜별 일정과 장소 보관함을 배치한다.
- 지도는 같은 화면에서 확대·축소하며, `?map=expanded` query로 확대 상태를 공유할 수 있다.
- 기존 `/trips/:tripId/map`은 삭제하지 않고 확대된 일정 화면으로 redirect해 이전 링크를 보존한다.
- 지도 입력 계약과 `MapAdapter` 경계는 바꾸지 않으며 실제 NAVER SDK는 장소·일정·지도 담당이 연결한다.
