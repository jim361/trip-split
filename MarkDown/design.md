# Trip Split Design Draft

## 1. 디자인 방향

Trip Split의 디자인은 Airbnb 스타일을 1순위 참고로 삼는다. 여행 맥락과 가장 잘 맞고, 따뜻한 코랄 포인트, 사진/장소 중심 구성, 둥근 UI가 여행 일정 앱에 자연스럽게 어울린다.

getdesign.md 기준 Airbnb는 "Travel marketplace. Warm coral accent, photography-driven, rounded UI."로 소개된다. Trip Split은 이 감성을 그대로 복제하지 않고, 여행 계획과 정산을 다루는 도구형 UI에 맞게 더 밀도 있고 실용적으로 적용한다.

## 2. 후보 디자인 비교

### 1순위: Airbnb

적합한 이유:

- 여행 서비스라는 도메인 일치
- 장소 카드, 일정 카드, 공유 흐름에 잘 맞음
- 코랄 포인트 컬러가 일정 핀/CTA에 사용하기 좋음
- 둥근 입력창과 카드가 모바일 PWA에 자연스러움

사용 범위:

- 전체 제품 톤
- 장소 카드
- 여행 생성 화면
- 공유 코드/초대 링크 화면
- Google 계정 연결 안내
- 지도 핀 컬러 일부

### 2순위: Cal.com

적합한 이유:

- 일정과 시간표 UI에 강함
- 중립적이고 정돈된 생산성 앱 느낌
- 타임테이블, 시간 선택, 날짜 탭에 참고하기 좋음

사용 범위:

- 타임테이블
- 날짜별 일정 편집
- 시간 입력 컴포넌트

### 3순위: Airtable

적합한 이유:

- 구조화된 데이터 편집에 강함
- 장소/비용/참여자 테이블 UI에 참고하기 좋음
- 협업 도구 느낌이 있음

사용 범위:

- 장소 보관함
- 지출 목록
- 참여자 관리

최종 추천은 Airbnb를 기본 DESIGN.md로 가져오고, Cal.com과 Airtable은 특정 화면 참고로만 사용한다.

## 3. 디자인 참고 도구

Airbnb 구성 요소를 빠르게 비교할 필요가 있을 때만 프로젝트 루트에서 다음 명령을 선택적으로 사용할 수 있다.

```bash
npx getdesign@latest add airbnb
```

생성된 결과는 시각 참고 자료로만 사용한다. 이 문서의 모바일 앱 셸, 정보 구조, 한국어 문구와 접근성 결정이 우선이며 자동 생성 파일로 덮어쓰지 않는다.

## 4. 모바일 우선 PWA 앱 셸

기본 설계 폭은 390px이며 작은 모바일 화면에서 모든 핵심 흐름이 완결되어야 한다. PC에서는 같은 페이지를 넓은 패널로 확장하되 메뉴와 사용자 흐름을 새로 만들지 않는다. 기본 PC breakpoint는 1024px로 둔다.

### 홈

- 여행 만들기
- 공유 코드 또는 초대 링크로 입장
- 최근 여행 목록
- 익명 세션 상태와 선택적 Google 계정 연결 안내
- `.trip.json` 백업 복원과 데모 데이터 진입은 보조 메뉴로 제공

### 여행 앱 셸

여행에 입장하면 상단 앱 바, 현재 페이지 영역, 주요 내비게이션으로 구성한다.

- 모바일 상단 앱 바: 뒤로가기, 여행 이름, 동기화 상태, 더보기만 우선 표시. 날짜는 현재 일정·지도 문맥에 두고 참여자, 공유, 계정 연결은 더보기 또는 보조 시트에서 제공
- PC 상단 앱 바: 같은 여행 문맥과 action을 유지하면서 날짜, 참여자, 공유와 계정 연결을 펼쳐 표시
- 주요 메뉴: `일정·지도`, `정산`, `영수증`
- 모바일: 세 메뉴를 화면 하단에 고정하고 아이콘과 한국어 label을 함께 표시
- PC: 같은 세 메뉴와 순서를 유지한 확장 내비게이션을 사용하고, 남는 폭은 각 페이지의 보조 패널에 사용
- 현재 메뉴는 색상뿐 아니라 아이콘/배경/텍스트 굵기로 함께 구분
- 장소 보관함은 네 번째 메뉴가 아니다. 일정·지도 통합 화면 안의 바텀시트 또는 보조 패널로 연다.
- 앱 바와 내비게이션은 모든 세 페이지에서 동일하고, breakpoint가 바뀌어도 현재 페이지와 편집 상태를 유지한다.

### 일정·지도 통합 화면

모바일은 스크롤 방향이 명확한 세로 타임라인을 기본으로 한다.

- 페이지 상단에는 약 220px 높이의 지도 미리보기를 먼저 표시하고, 번호 핀과 날짜별 직선 동선을 보여준다.
- `지도 크게 보기`로 지도를 같은 페이지 안에서 확대하며, 확대 상태는 `?map=expanded` query로 공유할 수 있다.
- 상단에 가로 스크롤 날짜 선택을 두고 선택 날짜를 sticky 상태로 유지
- 일정 항목은 시간, 순서, 장소명, 메모를 세로 타임라인에 표시
- 장소 검색, 네이버 장소 링크 붙여넣기와 직접 입력은 `장소 보관함` 바텀시트에서 제공
- 장소를 일정에 추가하거나 순서를 바꾸면 번호와 지도 동선이 같은 순서로 갱신
- 빈 시간 추가와 장소 연결은 한 손으로 누르기 쉬운 하단 CTA 또는 각 시간대 action으로 제공

PC도 지도 미리보기를 전체 폭 상단에 두고, 그 아래를 왼쪽 `날짜 목록`, 가운데 `세로 타임라인`, 오른쪽 `장소 보관함`의 3열로 확장한다. 날짜 목록은 약 200~240px, 장소 보관함은 약 320~360px로 두고 타임라인이 남은 폭을 사용한다. 실제 도로 경로와 예상 이동 시간은 표시하지 않는다.

### 정산 화면

모바일은 사용자가 자기 소비를 먼저 이해한 다음 송금 결과와 전체 지출을 확인하도록 아래 순서를 고정한다.

1. 개인 요약
   - `내가 결제한 금액`
   - `내가 부담한 금액`
   - 양수면 `받을 금액`, 음수면 `보낼 금액`, 0이면 `정산 완료`
   - 카테고리별 개인 소비 합계
   - 날짜·장소·메뉴 또는 지출 항목별 개인 소비 내역
2. 최종 정산
   - 누구에게 얼마를 보내거나 받아야 하는지 표시
   - 정산 문구 복사 action 제공
3. 지출 목록
   - 결제자, 총액, 내 부담액, 분할 방식, 연결된 장소/영수증 표시
   - 지출 추가, 수정과 삭제 제공

PC에서는 `지출 관리`, `개인 소비 요약`, `최종 정산 결과`를 다중 열로 배치한다. 지출 목록과 편집 영역이 가장 넓은 폭을 사용하고, 요약과 송금 결과는 스크롤 중에도 비교할 수 있는 side column으로 둔다. 정보 밀도는 높이되 금액 label을 생략하지 않는다.

### 영수증 화면

영수증은 `이미지 선택 → OCR 처리 → 항목 검토·분할 → 합계 확인 → 지출 저장` 순서로 진행한다.

- 사용자가 기기에서 이미지를 선택하면 로컬 미리보기만 표시하고 MVP에서는 Firebase Storage에 영구 저장하지 않음
- OCR 항목은 이름과 금액을 바로 수정할 수 있고 누락된 메뉴, 할인, 봉사료, 기타 조정을 수동 추가할 수 있음
- 각 항목에 `일반 항목`, `할인`, `봉사료`, `기타 조정` kind를 표시
- 영수증 전체 균등 분할, 항목별 소비자 지정, 참여자별 금액 직접 입력을 전환할 수 있음
- 공용 메뉴는 소비자 chip을 여러 명 선택하고 `균등 분할`을 적용
- 직접 입력에서는 참여자별 금액과 남은 미배분 금액을 동시에 표시
- 항목 합계와 영수증 총액, 참여자별 배분 합계가 맞지 않으면 저장 CTA 가까이에 오류와 차액을 표시
- OCR을 읽지 못하면 `총액으로 직접 등록`을 제공해 결제자, 총액, 소비자와 분할 방식만으로 저장 가능
- 사용자가 검토를 완료하기 전에는 정산 데이터로 저장하지 않음

모바일은 한 항목씩 읽기 쉬운 편집 카드와 sticky 저장 CTA를 사용한다. PC는 왼쪽 이미지 미리보기, 가운데 항목 편집, 오른쪽 배분·합계 요약의 다중 열로 확장한다.

## 5. 컬러 방향

기본은 밝고 따뜻한 여행 도구 느낌으로 간다.

```text
Background: #FFFFFF
Surface: #F7F7F7
Text: #222222
Muted Text: #717171
Line: #DDDDDD
Primary Coral: #FF385C
Primary Dark: #222222
Success: #00A86B
Warning: #F59E0B
Map Day 1: #FF385C
Map Day 2: #222222
Map Day 3: #007A87
```

네이버 지도 위에서는 핀과 동선이 잘 보여야 하므로 색상 대비를 우선한다.

## 6. 컴포넌트와 반응형 원칙

- 카드 반경은 12px 이하로 유지하고 중첩 카드는 피한다.
- 주요 CTA는 코랄 컬러를 사용하되 위험한 삭제 action과 구분한다.
- 지도 위 컨트롤은 흰색 배경, 얇은 라인과 충분한 그림자를 사용한다.
- 장소, 일정, 지출과 영수증 항목은 반복 카드 또는 명확한 구분선이 있는 리스트로 표현한다.
- 모든 touch target은 최소 44×44px로 하고 action 사이에 충분한 간격을 둔다.
- hover만으로 의미나 action을 노출하지 않는다. 아이콘 단독 버튼에는 접근 가능한 한국어 label을 제공한다.
- 금액은 우측 정렬, `tabular-nums`, 원 단위 천 단위 구분을 사용하고 `결제`, `부담`, `받기`, `보내기` label을 항상 함께 표시한다.
- 바텀시트는 drag handle, 제목, 닫기 action을 제공하고 키보드 focus가 시트 밖으로 빠지지 않게 한다.
- 모바일 입력 화면은 가상 키보드가 열려도 현재 필드와 저장 CTA를 가리지 않도록 scroll into view를 적용한다.
- PC에서는 같은 컴포넌트를 열과 패널로 재배치하며 모바일에 없는 필수 action을 추가하지 않는다.

### Safe area와 PWA viewport

- 앱 셸 높이는 `100vh` 대신 동적 viewport인 `100dvh`를 우선 사용한다.
- 상단 앱 바에는 `env(safe-area-inset-top)`, 하단 내비게이션과 sticky CTA에는 `env(safe-area-inset-bottom)`을 더한다.
- 스크롤 콘텐츠 하단에는 하단 내비게이션과 sticky CTA 높이만큼 여백을 확보해 마지막 항목이 가리지 않게 한다.
- 지도와 바텀시트도 같은 safe area를 공유하며 이중 padding을 만들지 않는다.
- standalone PWA, 브라우저 탭, 가로 회전에서 앱 바·내비게이션·바텀시트가 잘리지 않는지 확인한다.

## 7. 타이포그래피

- 기본 폰트: Pretendard, system-ui, sans-serif
- 숫자와 금액은 tabular-nums를 사용한다.
- 화면 제목은 크되 과하게 마케팅 페이지처럼 만들지 않는다.
- 서비스 첫 화면은 랜딩 페이지보다 바로 여행을 만들 수 있는 앱 화면에 가깝게 만든다.

## 8. 상태 디자인

모든 주요 화면은 다음 상태를 가진다.

- 빈 상태
- 로딩 상태
- 에러 상태
- 저장 중 상태
- 저장 완료 상태
- 동기화 상태
- 오프라인 상태
- 익명 세션 상태

상태는 색상만으로 구분하지 않고 아이콘, 짧은 한국어 문구와 가능한 다음 action을 함께 제공한다.

- 초기 로딩: 화면 골격을 유지하는 skeleton을 사용하고 전체 화면 spinner 남용을 피함
- 저장 중: `저장 중…`
- 저장 완료: `저장됨`을 짧게 표시한 뒤 동기화 상태로 전환
- 실시간 동기화 중: `변경사항 동기화 중…`
- 동기화 완료: `모든 변경사항이 저장됐어요`
- 오프라인: `오프라인이에요. 연결한 뒤 다시 시도해 주세요.` MVP는 정적 앱 셸 표시 외에 오프라인 편집·동기화를 보장하지 않음
- 재시도 가능한 오류: 원인 요약과 `다시 시도` 제공
- 권한 오류: `이 여행을 편집할 권한이 없어요`와 이전 화면 action 제공
- OCR 처리 중: 원본 미리보기를 유지하고 `영수증을 읽고 있어요…` 표시
- OCR 실패: `영수증을 읽지 못했어요`와 `다시 시도`, `총액으로 직접 등록` 제공
- 합계 불일치: `항목 합계가 영수증 총액과 1,000원 달라요`처럼 정확한 차액 표시

예시:

- 장소가 없을 때: "장소 링크를 붙여넣거나 검색해서 첫 장소를 추가하세요."
- 일정이 없을 때: "시간을 추가하고 장소를 연결해보세요."
- 지출이 없을 때: "첫 지출을 추가하면 정산 결과가 계산됩니다."
- 개인 소비가 없을 때: "아직 내가 부담한 지출이 없어요."
- 받을 금액과 보낼 금액이 모두 0원일 때: "정산이 모두 끝났어요."
- 익명 사용자일 때: "Google로 연결하면 내 여행 목록을 안전하게 보관할 수 있어요."

## 9. 피해야 할 것

- 과한 랜딩 페이지
- 어두운 대시보드 중심 디자인
- 지도보다 카드가 더 눈에 띄는 구성
- 복잡한 관리자 화면 같은 테이블
- 네이버 지도와 충돌하는 과한 배경색
- 저장 리스트 import에 의존하는 UX
- Google 로그인 없이는 사용할 수 없는 UX
- 동기화 상태를 숨기는 UX
- 장소 보관함을 별도 메인 메뉴로 만드는 UX
- 모바일과 PC에서 서로 다른 메뉴명·순서·URL을 사용하는 UX
- 개인 정산에서 결제액, 부담액과 송금 결과를 하나의 금액으로 섞어 표시하는 UX
- 합계가 맞지 않는 OCR 결과를 확인 없이 저장하는 UX
- MVP 영수증 이미지를 Firebase Storage에 영구 보관하는 UX

## 10. 디자인 결정

1차 디자인 결정은 다음과 같다.

- 기본 디자인 참고: Airbnb
- 일정 편집 참고: Cal.com
- 구조화된 목록 참고: Airtable
- 전체 톤: 밝고 따뜻한 여행 도구
- 핵심 화면: 여행 상세 화면
- 앱 셸: 390px 모바일 우선, 세 개의 공통 내비게이션, 1024px부터 같은 페이지를 다중 열로 확장
- 핵심 시각 요소: 장소 카드, 세로 타임라인, 번호 핀, 직선 동선, 결제액·부담액·정산 결과가 분리된 개인 요약
- 장소 보관함: 일정·지도 안의 바텀시트 또는 보조 패널
- 영수증 UX: OCR 초안을 항목별로 수정·배분한 뒤 저장하며 실패 시 총액 수동 등록
- 계정 UX: 기본은 익명 세션, 선택적으로 Google 계정 연결
- 협업 UX: 공유 코드로 입장한 참여자가 실시간으로 같은 여행을 편집
